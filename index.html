🎯 The Product Strategy

Simplicity vs. Feature Parity

At a senior level, my role was to resist the most dangerous force in product design — feature creep.

The Strategic Tension

Stakeholders wanted to match enterprise tools (Drift, Intercom) feature-for-feature. My research showed our users needed the opposite — they needed to create value in under 5 minutes or they'd leave forever. A complex feature set would kill activation.

The Strategic Decisions

The Problem Power users wanted deep customisation; SMB owners needed zero-friction setup.
The Decision Progressive disclosure — hide complexity behind an "Advanced" mode. New users get a guided template experience; power users unlock deeper logic when ready.

How it Works

A drone-based inspection app designed to work in harsh field conditions — gloves, bright sun, spotty connectivity — with zero tolerance for confusion or error.

01

Centralised Inspector Dashboard Single Source of Truth

Consolidates all tower data, inspection history, and maintenance schedules in one view — eliminating the need to switch between multiple systems and reducing pre-inspection planning time by 60%.

AI-Enhanced Defect Detection

Pre-configured report templates with auto-populated data cut generation from 2 hours to 15 minutes. Severity-based maintenance scheduling ensures critical issues get same-day decisions — reducing tower downtime by 35%.

🎯 The Product Strategy

Safety-First, Simplicity Always

Designing for the field is a completely different discipline to designing for the office. Every decision had one filter: "Can an inspector use this with gloves on, under direct sunlight, in a remote location?"

The Core Design Tension

Engineers wanted comprehensive data capture. Inspectors needed minimal cognitive load. I resolved this with a tiered information architecture — critical controls always visible on screen, secondary data one tap away, detailed analytics accessible from base camp only.

The Key Decisions

AI Transparency Inspectors wouldn't trust AI defect flags without explanation. I designed visual overlay annotations showing exactly what the AI detected and why.
Offline-First Architecture Field locations have zero connectivity. All core inspection workflows were designed to function fully offline, syncing when connection was restored.
🔍 The Journey

How I Got There

⚡

The Challenge & Pain Points

Inspectors risked their lives climbing 40-metre towers or spent thousands per tower on helicopter fly-overs. When they did spot a defect manually, it was usually already critical.

"Every time we climb those towers, we're putting our lives at risk. One wrong move and it's over."— Tower Inspector
"Helicopter inspections cost us thousands per tower, and we have hundreds to inspect annually."— Maintenance Manager
"By the time we spot a defect during manual inspection, it's often already critical."— Field Inspector
💡

The "Aha!" Moment

During contextual observation sessions embedded with inspectors in the field, I discovered the real problem wasn't just how they inspected — it was what happened after.

Defect reports were handwritten in the field, then manually transcribed back at base. Critical findings could sit unactioned for days. The insight: the reporting workflow was just as broken as the inspection workflow.

This led directly to the Template-Based Reporting System and same-day Maintenance Prioritisation Engine.

What I Learned

  • AI transparency through visual overlays — Showing inspectors exactly what the AI detected and why turned sceptics into advocates

The breakthroughs on this project — the AI overlay system, the offline-first architecture, the 3-click reporting flow — all came directly from field observation, not desk research. Design for the hardest context first; everything else gets easier.

Rocket Uniface UX

Building a Modern Open UX Widget Framework on Microsoft Fluent

65% Faster Development & $15K–$50K Annual Licensing Eliminated

I led the UX strategy and design system architecture for Rocket Uniface — an enterprise low-code platform. By pivoting from a "Closed Widget Library" to an "Open UX Widget Framework" built on Microsoft's Fluent Design System, we eliminated all third-party licensing costs, reduced development time by 65%, and repositioned Uniface competitively against OutSystems, Mendix, and Power Apps.

🚀 The Final Solution

How it Works

A comprehensive open UX Widget Framework built on Microsoft Fluent Design System — giving enterprise developers both a ready-made modern component library and the full flexibility to create custom widgets for any requirement.

01

Open UX Widget Framework Core Differentiator

A custom communication layer between web components and Uniface. Developers can create custom widgets using any web component library — not just the built-in ones — while maintaining full platform consistency.

Critical Pivot: From closed library → open framework. This became our #1 competitive differentiator.
02

Fluent-Based Widget Library Zero Licensing Cost

Pre-built widgets leveraging open-source Microsoft Fluent Web Components — eliminating the need for paid third-party libraries, resolving CSS style leakage, and providing a cohesive modern UI out of the box.

03

Flexible Style Adaptation

Fluent UI as the default, but every customer can apply their own brand style guide on top. Brand consistency and platform consistency — without compromise.

04

Redesigned IDE Navigation

Repositioned components (40% less mouse travel), simplified IDE navigation from 5 clicks to 2, enhanced documentation reducing onboarding from 4 hours to 45 minutes.

The Result: 85% faster developer onboarding
🎯 The Product Strategy

The Closed → Open Framework Pivot

The biggest strategic decision of the project happened mid-way through — and it changed everything.

The Mid-Project Pivot

We initially designed a closed widget library — a curated set of Fluent components. Mid-project testing revealed a critical gap: enterprise customers had unique UI requirements that no library could anticipate. They needed to create their own custom widgets. I pivoted the architecture to an open framework, transforming the product from a component library into a scalable platform.

The Strategic Tension

The Problem Standardisation vs. customisation. Enterprise customers need brand consistency but also unique components for their specific workflows.
The Decision Open architecture with Fluent as default layer. Customers get instant modern UI out of the box, plus the full ability to extend or replace any component with custom web components.
